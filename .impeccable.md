## Design Context

### Users
Pokopedia is for players planning Pokemon Pokopia while they play on Nintendo Switch. The primary context is desktop planning, with a secondary goal of adapting cleanly to phone-sized use later. Users want quick reference lookups, ownership tracking, habitat component checks, and help deciding which Pokemon should share a house.

### Brand Personality
Cozy, precise, and constructive. The app should feel like a field notebook that learned to be a sharp reference tool: warm enough for play, structured enough for serious planning, and practical enough to keep open beside the game.

### Aesthetic Direction
Desktop-first app workspace with a tactile field-guide tone: soft paper-like surfaces, crisp indexed lists, visible data density, and image-led Pokemon/habitat references. It should not feel like a marketing page, generic admin dashboard, neon game overlay, or bare Vite starter. Use a light theme by default because the app is likely used during planning sessions next to the Switch and needs easy scanning.

### Design Principles
1. Start with the useful surface, not an intro: search, track, compare, and plan should be available immediately.
2. Preserve reference clarity: source data, habitat requirements, favorites, and spawn conditions should stay easy to inspect.
3. Make planning explainable: optimizer results must show why a house group is good, not just output names.
4. Keep user data portable: localStorage is the working store, with import/export JSON for backup.
5. Design for desktop density first while keeping component structure ready for mobile adaptation.
